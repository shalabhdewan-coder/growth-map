## Verdict on current knobs/generators — dead-flag / narrow-reach audit

`LEVELS.sarah.maths[4] = {n:'Decimals & money', wm:12, mul2:1, dec:1, s:3, bar:'cmp', patt:'geo', chal:3}`.

Traced every reference to `L.dec` across all five generators before making any change. Result: **this is the most severe case of the narrow-reach bug class found so far on the ladder** (worse than rung 2's `mul2` gap or rung 4's `m_mistake` gap, both of which were confined to a single block). `L.dec` was checked in exactly **one** place in the entire codebase — the third `else if` in `m_words`' first special-problem branch (one milk-and-bread addition word problem). Every other block on the sheet ignored the rung's headline flag completely:
- **`m_warm`**: the `specials` array only ever checked `frac`/`longmul`/`mul2` — `dec` was never added as an option, so **none of the six warm-up items could ever be decimal arithmetic**, on the rung named "Decimals & money."
- **`m_words`' second block** (the `L.s>=3` steps-problem, shared with rungs 3/4/6/7/8/9): fixed to whole-number "stickers in boxes" regardless of `L.dec`.
- **`m_words`' third block** (the always-present change problem): item prices were always whole rupees (`R(6,19)` / `R(15,45)`), so even the "money" half of the rung's own name never produced a paise amount in this block.
- **`m_bar`**: no `dec` branch existed at all — `L.bar` was set to `'cmp'`, which produces a generic whole-number sticker-sharing bar model with zero money/decimal content. There was no decimal bar-model type to fall back to even if the knob had been set correctly.
- **`m_mistake`**: bank selection is `(L.chal>=3)?hard:easy`; rung 5's `chal:3` selects the `hard` bank, which (per rung 4's audit) contains a fraction-comparison item but **no decimal-place-value misconception at all**.

Net effect before this fix: across the whole six-block worksheet, `L.dec` produced exactly **one** decimal item out of roughly 15-20 individual questions on the sheet — everything else (warm-up ×6, steps-problem, change-problem, bar-model, mistake-spotting) was indistinguishable from a plain whole-number worksheet. This is precisely the bug class the task asked to hunt for: a flag reaching only one generator when the rung's entire premise implies it should touch several.

**Dead-flag masking check** (the rung-6 style bug, where one flag silently steals priority from another in the same if/else chain): `m_words`' first-block chain is `pct → ratio → frac → dec → algebra>=2`. Rung 5 sets none of `pct`/`ratio`/`frac`/`algebra`, so `L.dec` fires cleanly with no masking. `m_warm`'s `specials` array (now `frac`/`longmul`/`mul2`/`dec`/`ratio`/`pct`) picks uniformly among whatever's actually set — rung 5 has `mul2` and (after this fix) `dec` both present, and `P(specials)` gives each an equal chance per "special" roll, confirmed by repeated trials (see Verification). No masking found; the bug here was narrow reach, not priority-masking.

## Changes made in `growth-map.html`

1. **`m_warm`**: added `dec` to the `specials` array (`if(L.dec)specials.push('dec');` — same pattern used for `frac`/`longmul`/`mul2`) and a new `kind==='dec'` case: a decimal addition or subtraction to one decimal place, computed entirely in integer tenths (`Math.floor(t/10)+'.'+(t%10)`) to avoid any floating-point display bugs. This is additive-only — rungs that don't set `L.dec` never see `'dec'` in their `specials` array, so their warm-up behaviour is unchanged (verified below).
2. **`m_words`**:
   - Added a `fmt` helper (`p=>Math.floor(p/100)+'.'+String(p%100).padStart(2,'0')`) at the top of the function, working in integer paise throughout to avoid floating-point rounding errors — same pattern already used by the pre-existing decimal word problem in block 1.
   - Block 2 (the `L.s>=3` steps problem): added an `if(L.s>=3&&L.dec)` branch **before** the existing `if(L.s>=3)` branch, producing a 3-step decimal/money problem (start amount → two decimal subtractions → equal split among friends), built entirely in integer paise so the division is always exact. The pre-existing `else if(L.s>=3)` branch (whole-number stickers) is untouched and still fires byte-for-byte as before for every other `s>=3` rung that doesn't set `dec` (rungs 3, 4, 6, 7, 8, 9 — none of which set `dec`).
   - Block 3 (the always-present change problem): added an `if(L.dec)` branch producing a decimal-priced version (pencils at a paise-inclusive price, change computed in paise, rounded up to the nearest whole-rupee note). The `else` branch is the original whole-rupee code, unchanged, so every other rung is unaffected.
3. **`m_bar`**: added a new `t==='dec'` branch — a money bar-model ("total spent on two items, given one item's cost, find the other") — the first decimal/money bar-model type to exist in the codebase. Changed `LEVELS.sarah.maths[4].bar` from `'cmp'` to `'dec'` so rung 5 actually uses it. No other rung sets `bar:'dec'`, so this is purely additive.
4. **`m_mistake`**: added a `decBank` (two items: comparing 0.7 vs 0.65 by digit-count instead of place value, and 0.5+0.5="0.10" instead of 1.0 — both classic, well-documented decimal-place-value misconceptions) merged in via `if(L.dec)bank=bank.concat(decBank);`, following the exact same additive pattern rung 4's note established for `fracEasy`. Only rung 5 currently sets `L.dec`, so this has zero effect on any other rung.

No existing function signatures or return shapes changed; `ws_maths_sarah`'s call signature is untouched.
